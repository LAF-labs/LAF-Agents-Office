# LAF-Office 사용자 안내서

이 문서는 개발자가 아닌 팀원이 LAF-Office를 사용해 팀원과 함께 프로젝트를 관리하고, AI 에이전트에게 기획·문서화·개발 작업을 맡기고, 결과를 검토하는 흐름을 이해하기 위한 안내서입니다.

## 1. LAF-Office란?

LAF-Office는 스타트업이나 소규모 팀이 하나의 프로젝트 공간에서 사람 팀원과 AI 에이전트가 함께 일하도록 만든 프로젝트 워크스페이스입니다.

일반적인 협업 도구가 “할 일 목록”과 “대화방”에 머무른다면, LAF-Office는 다음까지 한 화면 안에서 연결합니다.

- 프로젝트 목표와 현재 우선순위 정리
- AI 에이전트와의 대화
- 프로젝트별 작업 생성과 상태 추적
- 팀 위키를 통한 결정 사항, 제약, 작업 이력 보관
- GitHub 저장소가 연결된 경우 브랜치, 코드 작업, PR 전달 결과 확인
- 사람 리뷰와 승인 흐름

핵심은 “AI에게 일을 시켰다”에서 끝나지 않고, 어떤 프로젝트의 어떤 작업을 누가 맡았고, 어떤 근거와 결과물을 남겼는지를 볼 수 있게 하는 것입니다.

## 2. 주요 개념

### 워크스페이스

워크스페이스는 팀이 함께 쓰는 가장 큰 단위입니다. 로그인 후 하나의 팀 공간에 들어오며, 여기서 프로젝트, 팀원, 에이전트, 설정을 관리합니다.

### 프로젝트

프로젝트는 LAF-Office에서 일을 묶는 기본 단위입니다. 예를 들어 “고객 포털 만들기”, “신규 랜딩페이지 개선”, “사내 자동화 구축”처럼 하나의 목표를 가진 작업 묶음입니다.

각 프로젝트에는 다음 정보가 붙습니다.

- 프로젝트 이름과 설명
- 현재 우선순위와 추가 메모
- 관련 GitHub 저장소 주소
- 프로젝트 작업 목록
- 프로젝트 전용 위키 문서
- 작업 진행과 전달 결과 이력

### AI 에이전트

기본 팀은 다음 역할로 구성됩니다.

- CEO: 목표를 정리하고 작업을 나누며 우선순위를 조율합니다.
- Frontend Engineer: 화면, 사용자 경험, 브라우저 인터랙션을 담당합니다.
- Backend Engineer: 서버 로직, 데이터 흐름, 실행 환경 연동을 담당합니다.
- Reviewer: 결과물의 품질, 누락, 회귀, 승인 기준을 검토합니다.

필요하면 Settings의 Agents 섹션에서 특정 도메인의 전문 에이전트 초안을 만들 수 있습니다.

### 작업 또는 티켓

작업은 프로젝트 안에서 실제로 처리할 단위입니다. 작업은 제목, 상세 설명, 담당자, 상태, 대화 스레드, 전달 결과를 가질 수 있습니다.

기본 상태는 다음과 같습니다.

- open: 아직 시작 전이거나 대기 중인 작업
- in_progress: 진행 중인 작업
- review: 검토가 필요한 작업
- blocked: 막힌 작업
- done: 완료된 작업
- canceled: 더 이상 진행하지 않는 작업

### 위키와 노트북

LAF-Office는 프로젝트 기억을 팀 위키에 남깁니다. 프로젝트 목표, 결정 사항, 제약, 작업 이력, 전달 결과가 위키에 쌓이므로 새 팀원이나 다른 에이전트가 맥락을 다시 읽을 수 있습니다.

에이전트별 노트북은 작업 중인 초안이나 관찰을 담는 공간입니다. 오래 보존할 가치가 있는 내용은 리뷰를 거쳐 팀 위키로 승격할 수 있습니다.

### Runner

Runner는 실제 코드 작업을 실행하는 로컬 실행기입니다. hosted workspace를 쓰는 경우에도 기획, 문서화, 작업 생성, 위키 관리는 Runner 없이 사용할 수 있습니다.

다만 AI 에이전트가 사용자의 컴퓨터나 지정된 머신에서 Codex CLI, Claude Code CLI, Git, GitHub 작업을 실행하려면 Runner 연결이 필요합니다.

## 3. 무엇을 할 수 있나?

### 프로젝트 관리

- 새 프로젝트를 만들고 프로젝트별로 작업을 정리합니다.
- 프로젝트 설명, GitHub 저장소, 추가 정보를 기록합니다.
- 작업을 상태별로 보고, 진행 중·검토 중·막힘·완료 상태를 확인합니다.
- 프로젝트별 대화와 작업 이력을 한곳에 모읍니다.

### AI 에이전트에게 업무 요청

- 자연어로 “이번 프로젝트에서 다음에 해야 할 일을 정리해줘”처럼 요청할 수 있습니다.
- 특정 에이전트를 `@ceo`, `@fe`, `@be`, `@reviewer`처럼 멘션해 부를 수 있습니다.
- 프로젝트를 선택한 상태에서 메시지를 보내면 해당 프로젝트 맥락이 함께 전달됩니다.
- `@all`을 사용하면 여러 에이전트에게 동시에 알릴 수 있습니다.

### 개발 작업 연결

GitHub 저장소와 Runner가 준비된 프로젝트에서는 에이전트가 개발 작업을 맡을 수 있습니다.

- 프로젝트 저장소 준비 상태 확인
- 작업별 브랜치 생성
- 코드 수정
- PR 생성
- PR 상태, 리뷰 결정, 체크 결과, 머지 가능 여부 기록
- 전달 결과를 작업과 프로젝트 위키에 남김

GitHub나 Runner가 아직 준비되지 않았더라도 프로젝트 기획, 문서화, 작업 쪼개기, 위키 정리는 계속 사용할 수 있습니다.

### 리뷰와 승인

작업이 review 상태가 되면 사람이 결과를 확인합니다. PR이나 전달 결과가 있으면 열어 보고, 필요하면 작업 스레드에 수정 지시를 남깁니다.

위키 승격이 필요한 노트북 항목은 Wiki의 Reviews 탭에서 승인하거나 수정 요청할 수 있습니다.

### 팀 지식 관리

- Wiki에서 프로젝트 문서와 팀 지식을 읽습니다.
- Notebooks에서 에이전트별 작업 초안을 확인합니다.
- Reviews에서 위키로 승격할 내용을 검토합니다.
- 검색으로 메시지, 위키, 노트북을 찾아봅니다.

### 팀원 초대

Settings의 Team 섹션에서 사람 팀원을 초대할 수 있습니다. 이메일 발송이 설정되어 있으면 초대 메일을 보내고, 그렇지 않으면 초대 링크를 복사해 직접 공유합니다.

## 4. 기본 사용 흐름

### 1단계: 가입 또는 로그인

처음 접속하면 로그인 또는 회원가입 화면이 나타납니다.

새로 시작하는 경우:

1. 이메일, 이름, 비밀번호를 입력합니다.
2. 새 팀을 만들거나 초대 토큰으로 기존 팀에 합류합니다.
3. 팀 이름을 정합니다.

이미 팀이 있는 경우:

1. 로그인으로 들어갑니다.
2. 초대를 받은 경우 초대 링크나 토큰으로 팀에 합류합니다.

### 2단계: 온보딩

첫 설정 화면에서는 프로젝트와 에이전트 팀을 준비합니다.

입력하는 내용:

- 회사 또는 프로젝트 이름
- 프로젝트 한 줄 설명
- 지금 가장 중요한 우선순위
- 기본 에이전트 이름
- 에이전트가 사용할 실행 환경
- 첫 작업 요청

이 단계에서 GitHub 연결은 필수가 아닙니다. 먼저 기획과 작업 정리부터 시작하고, 실제 코드 작업이 필요해질 때 연결해도 됩니다.

### 3단계: 프로젝트 만들기

Home 또는 Projects 화면에서 새 프로젝트를 만듭니다.

좋은 프로젝트 이름 예시:

- 고객 포털 MVP
- 랜딩페이지 전환율 개선
- 내부 정산 자동화
- 온보딩 이메일 개선

프로젝트를 만들면 프로젝트 전용 위키 문서도 함께 준비됩니다.

### 4단계: 에이전트에게 요청하기

Home 화면 하단 입력창에 요청을 씁니다.

예시:

```text
이번 프로젝트에서 먼저 해야 할 일을 5개 작업으로 나눠줘.
```

```text
@ceo 현재 목표를 기준으로 다음 스프린트 계획을 제안해줘.
```

```text
@reviewer 이 프로젝트 위키를 보고 빠진 의사결정이나 위험을 정리해줘.
```

프로젝트를 선택한 상태에서 메시지를 보내면 프로젝트 해시태그가 함께 붙어 해당 프로젝트 맥락으로 처리됩니다.

### 5단계: 작업을 만들고 추적하기

Projects 화면에서 프로젝트를 열고 새 티켓을 만듭니다.

작업을 만들 때 정하면 좋은 것:

- 무엇을 해야 하는지
- 왜 필요한지
- 완료 기준
- 맡길 에이전트 또는 사람
- 참고할 제약이나 링크

작업 상세 패널에서는 담당자, 상태, 상세 설명, 작업 대화를 확인하고 추가 지시를 남길 수 있습니다.

### 6단계: 개발 작업 맡기기

개발이 필요한 프로젝트라면 Projects 화면에서 GitHub 저장소를 연결합니다. 저장소 준비 상태가 통과되고 Runner가 연결되면 코딩 작업을 만들 수 있습니다.

코딩 작업은 일반적으로 다음 흐름으로 진행됩니다.

1. 사람이 작업을 만듭니다.
2. 에이전트가 작업을 맡습니다.
3. Runner가 로컬 또는 지정된 환경에서 작업을 실행합니다.
4. 코드 변경과 PR이 생성됩니다.
5. 작업에 전달 결과가 붙습니다.
6. 사람이 검토하고 완료 처리합니다.

저장소나 Runner 준비가 안 된 경우에도 작업은 기획 또는 문서 작업으로 만들 수 있습니다.

### 7단계: 검토하고 완료하기

작업이 review 상태가 되면 결과를 확인합니다.

확인할 것:

- 요청한 내용이 해결됐는지
- 전달 URL 또는 PR이 있는지
- PR 체크가 통과했는지
- 리뷰에서 수정 요청이 없는지
- 프로젝트 위키에 중요한 결정과 결과가 남았는지

문제가 있으면 작업 채팅에 수정 요청을 남기고, 괜찮으면 완료 처리합니다.

## 5. 화면 구성

### 전체 레이아웃

LAF-Office의 기본 화면은 다음 구조입니다.

- 왼쪽 사이드바: 주요 앱 이동, 프로젝트 목록, 워크스페이스 요약, 사용량
- 중앙 작업 영역: 현재 선택한 앱 또는 프로젝트 화면
- 상단 헤더: 현재 화면 제목, 검색, 로그아웃
- 런타임 스트립: 활성 에이전트와 막힌 작업 상태
- 오른쪽 패널: 에이전트 정보, 실시간 로그, 스레드 상세
- 하단 상태바: 현재 앱/채널, 모드, 에이전트 수, 연결 상태, 실행 제공자

### Home

Home은 가장 빠르게 프로젝트를 선택하고 에이전트와 대화하는 화면입니다.

주요 구성:

- 프로젝트 카드 목록
- 새 프로젝트 만들기
- 최근 대화 흐름
- 메시지 입력창
- `@agent` 자동완성
- 선택된 프로젝트와 멘션 대상 표시

일상적인 질문, 기획 요청, 다음 작업 정리는 Home에서 시작하면 됩니다.

### Projects

Projects는 프로젝트 운영의 중심 화면입니다.

주요 구성:

- 전체 프로젝트 목록
- 프로젝트별 상태와 티켓 수
- 프로젝트 정보 패널
- GitHub 저장소 연결과 준비 상태
- Runner 상태
- 작업 목록 또는 보드
- 새 티켓 만들기
- 티켓 상세 패널
- 티켓별 에이전트 채팅
- 전달 결과와 PR 상태

프로젝트를 실제로 관리하고 추적할 때는 Projects를 사용합니다.

### Wiki

Wiki는 팀 지식과 프로젝트 기억을 보는 화면입니다.

상단 탭:

- Wiki: 팀이 공유하는 공식 문서
- Notebooks: 에이전트별 작업 노트와 초안
- Reviews: 위키로 승격할 내용의 검토 큐

Wiki 화면 안에는 문서 목록, 본문, 관련 정보, 편집 이력, 감사 및 점검 화면이 포함됩니다.

### Settings

Settings는 워크스페이스 설정을 관리하는 화면입니다.

주요 섹션:

- General: 언어, 기본 에이전트, 실행 제공자, 동시 실행 수
- Agents: 기본 에이전트 확인과 전문 에이전트 초안 생성
- Team: 팀원, 역할, 초대 링크 관리
- Company: 회사 또는 프로젝트 맥락 입력
- Runner: 이 컴퓨터를 실행기로 연결
- API Keys: 필요 시 제공자 키 입력
- Danger Zone: 재설정 또는 워크스페이스 삭제

### 에이전트 패널

에이전트를 클릭하면 오른쪽에 에이전트 패널이 열립니다.

확인할 수 있는 것:

- 에이전트 이름과 역할
- 현재 상태
- 사용 중인 제공자
- 현재 작업
- 실시간 실행 로그
- 최근 활동
- 1:1 에이전트 스레드 열기

### 검색과 도움말

상단 검색 버튼으로 메시지, 위키, 노트북을 찾을 수 있습니다. 하단 상태바의 단축키 도움말을 열면 현재 화면에서 사용할 수 있는 키보드 조작을 확인할 수 있습니다.

## 6. 비개발자를 위한 사용 팁

### 요청은 “목표, 배경, 완료 기준”을 함께 적기

나쁜 예:

```text
이거 해줘.
```

좋은 예:

```text
@ceo 신규 고객 온보딩 프로젝트를 시작하려고 해.
목표는 첫 가입 후 10분 안에 핵심 기능을 써보게 만드는 거야.
이번 주에 할 수 있는 작업 5개로 나누고, 개발이 필요한 것과 문서만 필요한 것을 구분해줘.
```

### 프로젝트를 먼저 선택하고 대화하기

프로젝트를 선택한 상태에서 요청하면 대화와 작업이 프로젝트 맥락에 묶입니다. 나중에 위키와 작업 이력을 찾기 쉬워집니다.

### 작업 하나에는 결과 하나가 보이게 만들기

작업을 너무 크게 만들면 결과 검토가 어렵습니다. “전체 서비스 개선”보다 “회원가입 화면의 오류 메시지 개선”처럼 확인 가능한 단위로 나누는 것이 좋습니다.

### 중요한 결정은 위키에 남기기

채팅은 흐름을 남기고, 위키는 오래 볼 결정을 남기는 곳입니다. 정책, 목표, 제약, 고객 약속, 승인된 방향은 위키에 남겨야 다음 작업에서 재사용됩니다.

### 코딩 작업은 준비 상태를 먼저 확인하기

AI가 코드까지 수정하려면 GitHub 저장소와 Runner가 준비되어야 합니다. 준비 전에는 기획, 문서화, 작업 분해 요청부터 진행하세요.

## 7. 자주 묻는 질문

### 개발자가 아니어도 쓸 수 있나요?

네. 프로젝트 만들기, 작업 요청, 에이전트와 대화, 위키 확인, 결과 검토는 비개발자도 사용할 수 있습니다. 코드 실행과 PR 생성에는 GitHub와 Runner 준비가 필요하지만, 그 전 단계의 기획과 관리 기능은 그대로 사용할 수 있습니다.

### GitHub를 꼭 연결해야 하나요?

아닙니다. GitHub는 코드 작업을 맡길 때 필요합니다. 저장소를 연결하지 않아도 프로젝트 관리, 대화, 작업 생성, 문서화, 위키 관리는 가능합니다.

### Runner를 설치하지 않으면 무엇이 제한되나요?

Runner가 없으면 로컬 Codex/Claude 실행, Git 체크아웃, 코드 수정, PR 생성 같은 실제 개발 실행이 대기합니다. 대신 기획, 작업 분해, 위키 정리, 팀원 초대, 프로젝트 관리는 사용할 수 있습니다.

### AI 에이전트가 한 일을 어디서 확인하나요?

Projects의 작업 상세, 전달 결과, PR 링크, 작업 채팅, Wiki의 프로젝트 문서, 에이전트 패널의 실시간 로그와 최근 활동에서 확인합니다.

### 팀원이 같이 쓸 수 있나요?

네. Settings의 Team에서 초대 링크를 만들거나 이메일 초대를 보낼 수 있습니다. 합류한 팀원은 같은 워크스페이스에서 프로젝트와 작업을 함께 봅니다.

### CRM, 이메일, 캘린더처럼 외부 업무 도구도 대체하나요?

현재 기준으로 LAF-Office의 중심은 프로젝트, 작업, AI 에이전트 실행, 위키 기억입니다. CRM, 이메일, 캘린더, 알림 같은 hosted integration 상태는 기본 전제로 두지 않습니다.

## 8. 한 줄 요약

LAF-Office는 프로젝트별로 사람과 AI 에이전트가 함께 계획하고, 작업을 나누고, 개발 결과를 전달받고, 그 과정을 위키에 남기는 팀용 AI 프로젝트 워크스페이스입니다.
