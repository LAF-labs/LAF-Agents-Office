# Agent Memory Packets

LAF-Office treats the markdown wiki as shared memory for agents first. Humans
can read it, but the primary runtime path is:

```
task -> scoped memory packet -> agent action -> durable write-back
```

An agent memory packet is a compact, task-scoped contract generated by the
broker. It tells the agent which wiki context is already loaded, which sources
are canonical, what constraints must be obeyed, where to start, and how to
write useful state back.

## v1 Shape

`agent-memory/v1` is intentionally small:

- `task`: the task identity, status, owner, channel, execution mode, and
  worktree fields.
- `project`: project identity, canonical wiki path, repo boundary, lead, and
  description when the task belongs to a project.
- `must_read`: canonical memory sources for the task.
- `loaded_context`: which excerpts are already injected into the work packet.
- `decisions`: durable decisions extracted from the project wiki.
- `risks`: active risks/blockers extracted from the project wiki.
- `open_questions`: unresolved questions extracted from the project wiki.
- `recent_work`: recent project task receipts, delivery URLs, and blockers.
- `must_obey`: rules the agent must honor before acting.
- `start_here`: first steps that prevent broad, wasteful rediscovery.
- `write_back`: how the agent should preserve new knowledge.
- `unavailable`: memory sources that could not be loaded.

The packet does not replace markdown. It is a runtime index over markdown, task
state, and project state.

## Current Integration

Project tasks now receive:

1. A JSON `agent-memory/v1` contract in the task execution packet.
2. The project wiki excerpt immediately below that contract.
3. Extracted decisions, risks, open questions, and recent project work receipts.
4. A reload path at `GET /tasks/context?id=<task_id>`.
5. An MCP tool, `team_task_context`, for agents to reload the same packet.

The project wiki remains the source of truth. The packet is generated from task
state plus `team/projects/{project_id}.md`; if the article is missing, the
broker materializes it before building the packet.

New project wiki pages include `Decisions`, `Risks`, and `Open questions`
sections so agents have stable write-back targets. Boilerplate bullets in those
sections are ignored by the packet compiler; only project-specific bullets are
surfaced to future agents.

## Write-Back Rule

Agents should not write raw thoughts directly into canonical shared memory.

- Working notes go to `notebook_write`.
- Durable conclusions go through `notebook_promote`.
- Task lifecycle state goes through `team_task`; the broker appends project task
  events to the project wiki.
- Direct `team_wiki_write` is for explicit canonical edits, bootstrap/admin
  changes, or human-directed updates.

This keeps shared memory useful for future agents instead of turning it into a
chat transcript.
